\section{Structural observations}
\begin{enumerate}
\item \textbf{Sumcheck hot path: 46,718\,ms (\textasciitilde58\% of modular prove).} Modular's \texttt{InstanceSegmentedReduce + InstanceBind + InstanceReduce} trio replaces what core does inside compiled \texttt{prove\_stageN} closures. The combined overhead vs. core's same-shape sumchecks is \textasciitilde 20\texttimes{}. Every round, modular dispatches through \texttt{CpuBackend} kernels that core inlines directly. See the call-count explosion table.
\item \textbf{Derived-polynomial materialization: pm::Derived = 14,638\,ms.} RAM witness polys alone ($\texttt{ram\_val} + \texttt{ram\_ra\_indicator} + \texttt{ram\_combined\_ra}$) cost 14,424\,ms. Core materializes equivalent polynomials once up-front in \texttt{generate\_and\_commit\_witness\_polynomials} (\textasciitilde 819\,ms). Modular pays this per-use, not per-poly. If these polys are re-derived on each \texttt{Bind} call, caching or bulk-derivation is a big target.
\item \textbf{Materialize/EqProject subsystem: 25,219\,ms.} The \texttt{mb::EqProject + CpuBackend::eq\_project} path alone is 9,816\,ms self. Core's equivalent \texttt{EqPolynomial::evals\_parallel} is 9.0\,ms --- a \textasciitilde 1000\texttimes{} gap. Likely re-computing eq tables every round rather than binding incrementally.
\item \textbf{PCS commit+open is roughly parity.} Modular \texttt{Commit} = 2,975\,ms vs. core's committed polynomial generation = 819\,ms. Modular \texttt{Open} (1,623\,ms) is already comparable to core \texttt{create\_evaluation\_proof} (706\,ms). Dory is not the modular bottleneck; sumcheck and witness-derivation are.
\item \textbf{Verify delta is suspicious: core 83.8\,ms vs modular 0.8\,ms.} Modular's \texttt{jolt\_verifier::verify} is two orders of magnitude faster, suggesting the Dory PCS opening is \emph{not} being verified on the modular path. Confirm whether the native modular verify is pipelined against the Dory opening proof before declaring modular verify a win.
\item \textbf{Memory: 3.3\texttimes.} Modular peak RSS 5.16\,GB vs core 1.55\,GB. Likely sources: lazily-derived polys held in CpuBackend buffers + per-op output buffers not recycled + multiple copies of eq tables.
\item \textbf{Per-round dispatch: \texttt{interpolate\_inplace} runs 17,416 times in modular vs. not at all as a distinct span in core} (core inlines the interpolation inside its sumcheck compute\_message). Batching \texttt{interpolate\_inplace} across kernels in the same round is the prototypical amortization target.
\end{enumerate}
